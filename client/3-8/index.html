3-8 Script execution in browsers

In all previous example in this chapter, we put <script> elements near the end of an HTML document. The reason is that the document tree is (almost) complete when these <script> executes. Now, we’ll explain the rationale for this scheme and discuss the JavaScript execution in browsers.

<!doctype html>
...
<body>
  <p>content for building document tree</p><p>...</p>
  <!-- load external script library -->
  <script src="..."></script>
  <script>
    // embedded script
  </script>
</body>

Page load

Page loading refers to the steps that a browser performs to load the various components of a web page.

  • usually starts with the URL of an HTML page
  • the browser downloads the HTML document
  • while downloading the HTML document,
    • the parser builds the document tree (DOM tree) incrementally
    • it discovers links to other resources
    • when the parser encounters <script> elements, it executes the script immediately.
    • parsing pauses until <script> execution finishes or aborts because of errors
  • the browser downloads discovered resources in parallel
  • when the document tree is complete, and all resources discovered have been downloaded and processed, page load finishes.

Parsing is the algorithm to convert an HTML document (text file) into a document tree (tree-formed data structure). The parser is a part of the browser that performs parsing.

Event-driven programming

After page load, JavaScript code only runs as callbacks. There is an event loop that waits for events (mainly UI events from the document tree). When a UI event happens, the event loop runs any registered event handlers.

<button id=btn3>Go</button>
<script>
  function btn_click (e) {
    // do something when the button is clicked
  }

  $('#btn3').on('click', btn_click);
</script>

Page loading will run the <script> element to define the function and register the event handler. After page load, when the click event occurs at the button, the browser calls the event handler.

The other situation is a callback registered to handle I/O completion. When we discuss AJAX in Chap 5, you’ll see how to register a callback to handle response that a web server returns for an AJAX call.

An important thing to remember is that there is only 1 single thread to run JavaScript code in a browser pane (i.e. the window for an HTML document). This is sometimes described as ‘JavaScript is single-threaded’. This ensures that only 1 thread can make change to the document tree (no concurrency issues in document tree updates). When there are multiple events happening, the browser will run the relevant event handlers one by one.

So in general, you should write event handlers that return quickly. Avoid long running event handlers, and don’t do long computation in event handlers. Refer to example p384.html for the effect of a long running event handler.
